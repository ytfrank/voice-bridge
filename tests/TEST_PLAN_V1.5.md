## 一、需求理解与补充意见

### 当前理解
本次 V1.5 的发布门禁集中在 5 个方面：
1. **稳定性**：2 小时连续运行不崩
2. **恢复能力**：530 崩溃 3s 内恢复、来电中断 5s 内恢复
3. **识别质量**：英文 ASR 准确率 ≥ 97%，P90 延迟 ≤ 3s
4. **翻译体验**：中文区零跳动
5. **导出能力**：会话详情页支持导出 Markdown，含时间戳，走 iOS Share Sheet；历史会话详情不展示生词表

### 我的补充意见
目前有 3 个需要开发侧尽快确认的实现口径，否则会影响测试效率：
1. **530 模拟方式**：建议提供测试开关或明确 kill 进程是否等价于 530 场景
2. **来电恢复验证环境**：需要明确是真机验证还是模拟器可稳定复现 AudioSession interruption
3. **V1.4 对比环境**：若要做跳动对比，需确认 V1.4 可运行版本或保留录屏基线

## 四、五项重点测试设计

### 1）准确率测试

#### 测试素材
- 已有：`musk_21s_correct.wav`
- 需补充 2 段英文素材：
  - **素材A**：美式口音、正常语速、演讲/新闻类，15-30s
  - **素材B**：英式或非美式口音、较快语速/连读明显，15-30s

#### 素材要求
- 清晰人声，可确认 ground truth
- 无强背景音乐
- 便于逐句核对
- 三段素材合计覆盖：不同口音、不同语速、不同句型

#### 评估方法
- 主指标：**WER（Word Error Rate）**
- 计算：`准确率 = 1 - WER`
- 补充复核：专有名词、数字、国家名、人名等关键实体不得出现致命误识别

#### 验收标准
- 综合准确率 **≥97%**
- 单段音频不得出现明显灾难性误识别

#### 输出物
- `tests/assets/asr_samples.md`：素材清单 + ground truth
- `tests/report_accuracy.md`：逐句对比 + WER + 截图

---

### 2）稳定性测试（2 小时）

#### 目标
验证连续录音/转写/翻译 2 小时：
- 0 崩溃
- 短断可恢复
- 内存平稳
- PM2/WS 守护有效

#### 脚本要求
脚本应支持：
1. 持续发送音频或模拟连续输入
2. **每 30 分钟**记录一次发布门禁要求的状态快照
3. 同时建议每 30 秒采集一次运行指标（便于定位波动）
4. 记录：
   - 时间
   - 进程状态
   - WebSocket 状态
   - PM2 重启次数
   - CPU / 内存
   - 错误日志摘要
5. 结束后汇总：
   - 崩溃次数
   - 断线次数
   - 平均恢复时长
   - 最大恢复时长

#### 验收标准
- 2 小时内 **崩溃次数 = 0**
- 短断后 **30s 内自动恢复**
- 无人工重启
- 无明显内存泄漏趋势

#### 输出物
- `tests/scripts/soak_test.sh`
- `tests/stability/runtime_metrics_*.csv`
- `tests/report_stability.md`

---

### 3）翻译跳动量化评估

#### 跳动定义
**跳动** = 中文区已展示内容再次发生覆盖、改写、回退、重排。

#### 量化指标
1. **句子跳动率（主指标）**
   - 一句中文首次完整展示后，若又被修改，记为该句发生跳动
   - `句子跳动率 = 发生跳动的句子数 / 总句子数`

2. **平均改写次数（辅指标）**
   - `平均改写次数 = 总改写次数 / 总句子数`

#### 对比方式
- 同一段连续 2 分钟英文音频
- V1.4 与 V1.5 分别录屏
- 逐句回放统计：
  - 是否先显示半句后改写
  - 是否已显示内容回退
  - 是否同一句重复重写

#### 验收标准
- **V1.5 跳动率 = 0**
- **V1.5 平均改写次数 = 0**

#### 输出物
- `tests/report_translation_jitter.md`
- 对比录屏 / 截图证据

---

### 4）PM2 恢复验证

#### 场景 A：530 / 服务崩溃恢复
目标：验证 **3s 内自动恢复**

记录点：
- T0：故障注入时间
- T1：PM2 发现退出/重启时间
- T2：服务恢复可访问时间
- 恢复耗时 = `T2 - T0`

验收：
- 恢复耗时 **≤3s**
- PM2 有重启记录
- 日志可追溯故障原因

#### 场景 B：来电中断恢复
目标：验证 **5s 内录音自动恢复**

记录点：
- 中断发生时间
- UI 是否显示“录音已暂停/正在恢复”
- 中断结束后恢复开始时间
- 录音链路恢复时间

验收：
- 恢复耗时 **≤5s**
- UI 状态提示明确
- 中断前内容不丢失，恢复后链路连续

#### 输出物
- `tests/report_pm2_recovery.md`
- PM2 日志截图 / iOS 录屏截图

---

### 5）导出功能验证（新增）

#### 功能点
1. 会话详情页存在 **“导出 Markdown”按钮**
2. 点击后生成带时间戳 Markdown
3. 走 iOS Share Sheet，可分享/存入 Files
4. 历史会话详情页 **不展示生词表**

#### 验证点
**A. 按钮展示**
- 会话详情页入口可见
- 按钮命名正确、位置合理、可点击

**B. 导出文件格式**
验证导出内容是否符合规范：
```markdown
# voice-bridge 录音记录
**录音时间**：YYYY-MM-DD HH:MM:SS
**时长**：X小时 X分钟

---

[HH:MM:SS] **EN**: ...
[HH:MM:SS] **CN**: ...
```

重点检查：
- 标题正确
- 录音时间存在且格式正确
- 时长字段存在
- 每句 EN/CN 均带 `[HH:MM:SS]`
- 中英文时间戳对齐
- 文件名格式：`voice-bridge-YYYY-MM-DD-HH-MM.md`

**C. iOS Share Sheet**
- 点击导出可正常拉起 Share Sheet
- 可分享到 Files app
- 导出后文件可打开、内容完整

**D. 历史会话详情**
- 不展示生词表模块
- 仅展示英文字幕 / 中文翻译对应内容

#### 验收标准
- 所有验证点通过
- 无生词表残留展示
- Markdown 格式完全符合需求

#### 输出物
- `tests/report_export_markdown.md`
- 导出样例文件
- Share Sheet 截图 / 历史会话截图

## 五、测试执行顺序

### Phase A：提测前准备（与 Peter 并行）
1. 补齐 2 段 ASR 测试素材与 ground truth

### Phase C：专项验证
1. 准确率测试
2. 延迟测试
3. 翻译跳动对比

5. 来电恢复测试
6. 导出功能验证
7. 历史会话无生词表验证

### Phase D：发布门禁
1. 2 小时稳定性完成
2. 汇总专项报告
3. 输出 `tests/report.md`
4. 给出 Pass / Fail 结论

## 六、上线阻断标准

以下任一不满足，**不建议上线**：
- 2 小时稳定性未完成或未通过
- 530 恢复时间 > 3s
- 来电恢复时间 > 5s 或不能自动恢复
- ASR 准确率 < 97%
- 中文区跳动率 > 0
- 导出 Markdown 格式不符合要求
- iOS Share Sheet 不可用
- 历史会话详情仍展示生词表
