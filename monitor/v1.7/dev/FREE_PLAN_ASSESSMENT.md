## 0. 目标定义

根据波哥 2026-04-04 确认：

- **免费路线主方案**：广播/采访/一对一交流可用
- **轻生产级**：3~5 人稳定可用
- **"免费"的含义**：不用付费 ASR / 翻译 API，用开源/本地模型
- **核心场景**：
  1. 外部广播/视频/播客 → 手机收音 → 英文转写 → 中文翻译
  2. 面对面与外国人交流 → 手机收音 → 英文转写 → 中文翻译

---

## 1. 先讲结论

### 免费路线能做到什么

| 指标 | 安静环境（广播/采访） | 轻噪音（面对面交流） | 嘈杂/歌曲 |
|---|---|---|---|
| ASR 准确率 | **93% ~ 96%** | **90% ~ 95%** | **不稳定，60%~85%** |
| 端到端延迟 | **2 ~ 4s** | **2.5 ~ 5s** | 同 |
| 3~5人并发 | ✅ 可支撑 | ✅ 可支撑 | ✅ 但质量本身不稳 |
| 幻觉率 | <2% | <3% | 更高 |
| 空结果率 | <3% | <5% | 更高 |

### 免费路线做不到什么

- **稳定 97%+ 准确率**：在真实环境下（非实验室干净音频），开源模型很难稳定到 97%
- **稳定 <2s 延迟**：流式可以做到 partial 首词 <1s，但完整可用翻译输出通常 2s+
- **歌曲高准确率**：不建议作为核心承诺场景

### 最重要的判断

> **免费路线可以做到"广播/采访/一对一交流可用 + 3~5人轻生产"，**
> **但前提是必须做架构升级（流式），不是在当前 chunk 架构上修修补补。**

## 2. 推荐的免费技术栈

### 2.1 ASR 模型推荐

经过对比，推荐 **Whisper large-v3-turbo** 作为免费方案首选：

| 模型 | 参数量 | VRAM(int8) | 速度(GPU) | 准确率 | 推荐度 |
|---|---|---|---|---|---|
| Whisper medium（当前） | 769M | ~3GB | ~2x 实时 | 基线 95.56% | ⚠️ 当前用的，不够 |
| Whisper large-v3 | 1550M | ~5GB | ~1x 实时 | 最高 | ❌ 太慢，难并发 |
| **Whisper large-v3-turbo** | **809M** | **~3-4GB** | **~3x 实时** | **接近 large-v3** | ✅ **推荐** |
| Whisper small | 244M | ~1.5GB | ~6x 实时 | 明显更差 | ❌ 准确率不够 |

**为什么选 large-v3-turbo：**
- 参数量只有 large-v3 的一半（809M vs 1550M）
- 速度约 3 倍实时（GPU），远快于 large-v3
- 准确率接近 large-v3，明显优于 medium
- int8 量化后 VRAM ~3-4GB，单张 24GB GPU 可同时跑 3-5 路
- **100% 免费开源**，OpenAI 发布，Apache 2.0 许可

### 2.2 流式框架推荐

经过调研，推荐两个方案（二选一）：

#### 方案 A（更成熟）：WhisperStreaming
- Repo：`ufal/whisper_streaming`
- 论文实测：**3.3 秒延迟**（含 ASR + 流式策略）
- 支持 faster-whisper 后端
- 支持 VAD（语音活动检测）
- 支持 translation 任务（直接英文→其他语言）
- 3.6k Stars，学术验证

#### 方案 B（最新 SOTA）：SimulStreaming
- Repo：`ufal/SimulStreaming`
- WhisperStreaming 的升级版，2025 年发布
- **比 WhisperStreaming 快 ~5 倍**
- 赢得 IWSLT 2025 同声传译比赛
- 支持 Whisper large-v3 + 大语言模型翻译
- 需要更多 GPU 资源（推荐 1-2 张 GPU）

**我的建议：先用方案 A（WhisperStreaming），更成熟，资源需求更低，适合 3-5 人轻生产。**

### 2.3 翻译模型推荐

翻译这块有几个免费选项：

| 方案 | 模型 | 大小 | 速度 | 翻译质量(en→zh) | 推荐度 |
|---|---|---|---|---|---|
| **Whisper 内置 translate** | Whisper 自带 | 0（复用 ASR 模型） | 最快 | 中等（直接出英文，不翻中文） | ⚠️ 只能翻成英文 |
| **NLLB-200** | facebook/nllb-200-distilled-600M | 600M | 快 | 中高 | ✅ 推荐备选 |
| **Helsinki-NLP/opus-mt-en-zh** | 小型专用模型 | ~300M | 很快 | 中 | 可用 |
| **继续用 GLM-4-Flash** | 智谱 API | 0（API调用） | 中 | 高 | ⚠️ 注意：严格说不是"免费" |

**关于 GLM-4-Flash：**
- 智谱有免费额度，但生产级使用可能超出免费额度
- 如果波哥可以接受"翻译用极低成本的 API"（GLM-4-Flash 很便宜），翻译质量会明显更好
- 如果一定要 100% 免费，用 NLLB-200 做本地翻译

**我的建议：**
- **首选**：继续用 GLM-4-Flash 做翻译（成本极低，质量最好）
- **纯免费备选**：NLLB-200-distilled-600M（本地跑，不花钱，质量略降）

### 2.4 整体架构

```
手机 App
  │
  ├── 持续采集音频
  ├── WebSocket 长连接上传音频流
  │
  ▼
服务端
  │
  ├── Session Gateway（认证、session 管理、限流）
  │
  ├── VAD（语音活动检测，跳过静音段）
  │
  ├── Streaming ASR Worker Pool
  │   └── faster-whisper + large-v3-turbo (int8)
  │   └── WhisperStreaming 流式策略
  │
  ├── Translation Worker
  │   └── GLM-4-Flash API 或 NLLB-200 本地
  │
  ├── WebSocket 下推 partial/final 结果
  │
  ▼
手机 App 展示字幕
```

## 3. 部署方案：3~5 人可用需要什么

### 3.1 最小可行部署

**一台 GPU 服务器即可支撑 3~5 人。**

#### 推荐配置

| 项目 | 推荐 | 说明 |
|---|---|---|
| GPU | 1 张 24GB（如 RTX 4090 / L4） | large-v3-turbo int8 ~3-4GB/模型实例 |
| CPU | 8 核+ | 音频预处理、VAD、翻译 |
| 内存 | 32GB+ | 多 session 缓存 |
| 网络 | 稳定公网 IP + 低延迟 | WebSocket 需要 |
| 系统 | Ubuntu 22.04+ | CUDA 驱动 |

#### 为什么一张 GPU 够

- large-v3-turbo int8：每个模型实例 ~3-4GB VRAM
- 速度 ~3x 实时：处理 1 秒音频只需 ~0.33 秒
- 3-5 个流，通过时间复用（不是每秒都在处理），单 GPU 可以轮转
- VAD 会跳过静音段，实际 GPU 负载远低于理论最大值

### 3.2 成本估算

#### 方案 A：租用云 GPU

| 平台 | GPU | 价格 | 月成本（24h 在线） | 月成本（12h/天） |
|---|---|---|---|---|
| RunPod | RTX 4090 | ~$0.34/hr | ~$245/月 | ~$123/月 |
| Modal | L4 | $0.80/hr | ~$576/月 | ~$288/月 |
| Modal | A10 | $1.10/hr | ~$792/月 | ~$396/月 |

> **RunPod RTX 4090 是目前性价比最高的选项。**

#### 方案 B：自购硬件

| 设备 | 价格 | 说明 |
|---|---|---|
| 二手 RTX 4090 | ¥8,000~12,000 | 一次性投入，24GB VRAM |
| 配套主机 | ¥5,000~8,000 | CPU + 内存 + 电源 |
| 月电费 | ~¥200~400 | 取决于使用时长 |

> 自购硬件：一次性 ¥15,000~20,000，之后月成本仅电费。
> **如果长期运行，3~6 个月就比租云 GPU 便宜。**

#### 方案 C：Apple Silicon Mac（如果已有）

| 设备 | 说明 |
|---|---|
| Mac Studio M2 Max/Ultra（64GB+） | mlx-whisper 跑 large-v3-turbo |
| 预估并发 | 2~3 路（M2 Max），3~5 路（M2 Ultra / M4 Max） |

> 如果团队已有高配 Mac，可以先拿来跑原型验证。

### 3.3 翻译成本（如果用 GLM-4-Flash）

GLM-4-Flash 定价很低：
- 输入：¥0.1 / 百万 tokens
- 输出：¥0.1 / 百万 tokens

按 3~5 人持续使用 1 小时估算：
- 每人每分钟约 50~100 个翻译 token
- 5 人 × 60 分钟 × 100 token = 30,000 token/小时
- 成本：约 **¥0.003/小时**
- 即使每天用 8 小时：约 **¥0.024/天**

> **翻译成本几乎可以忽略不计。**

## 4. 免费路线的现实预估

### 4.1 场景 A：英文广播 / 采访 / 播客

这是免费路线**最有把握做好的场景**。

| 指标 | 预估值 | 说明 |
|---|---|---|
| ASR 准确率 | **93% ~ 96%** | 安静环境、清晰口语 |
| 端到端延迟 | **2 ~ 4s** | 流式 partial 更快，final 略慢 |
| Partial 首词延迟 | **<1s** | 用户可以很快看到"正在识别" |
| 翻译可理解度 | **高** | GLM-4-Flash 翻译质量不错 |
| 空结果率 | **<3%** | VAD 会过滤静音 |
| 幻觉率 | **<2%** | large-v3-turbo 比 medium 更稳 |

**体验描述**：
- 用户打开 App，对准广播/视频
- <1s 开始看到英文字幕滚动
- 2~4s 后看到对应中文翻译
- 大部分内容可准确理解
- 偶尔人名、专有词会丢或错

### 4.2 场景 B：面对面交流

| 指标 | 预估值 | 说明 |
|---|---|---|
| ASR 准确率 | **90% ~ 95%** | 取决于距离、噪音、口音 |
| 端到端延迟 | **2.5 ~ 5s** | 面对面更碎片化，延迟略高 |
| 翻译可理解度 | **中高** | 日常对话翻译质量 OK |
| 空结果率 | **<5%** | 说话间隔多，可能偶尔空 |

**体验描述**：
- 外国人说一句话
- 2.5~5s 后看到英文 + 中文
- 日常交流内容基本可理解
- 口音重或说话快时会掉准确率
- 多人同时说话时明显下降

### 4.3 场景 C：英文歌曲（不建议承诺）

| 指标 | 预估值 | 说明 |
|---|---|---|
| ASR 准确率 | **60% ~ 85%** | 伴奏干扰严重 |
| 翻译质量 | **不稳定** | 歌词翻译本身就难 |

**建议**：作为"额外能力"展示，不作为产品承诺。

## 5. 能承诺什么 vs 不能承诺什么

### ✅ 可以承诺

- 英文广播/采访/播客 → 英文字幕 + 中文翻译，**基本可用**
- 面对面交流 → 英文字幕 + 中文翻译，**日常场景可用**
- 3~5 人同时使用，**基本流畅**
- 端到端延迟 **通常 2~5 秒内**
- 翻译能帮用户**理解大意**

### ❌ 不能承诺

- 稳定 97%+ 准确率（真实环境做不到）
- 稳定 <2s 延迟（流式 partial 可以，但完整翻译做不到）
- 英文歌曲高准确率
- 多人同时说话场景
- 强噪声环境（工地、KTV、嘈杂餐厅）

### ⚠️ 有条件可以做到

- 安静环境下 **95%+ 准确率**（有把握）
- 3s 内出翻译结果（大部分情况可以，但不是 100%）

## 6. 与当前架构的差距

| 维度 | 当前 V1.6 | 免费升级版（目标） | 差距 |
|---|---|---|---|
| ASR 模型 | Whisper medium | large-v3-turbo | 模型升级 |
| 架构 | 5s chunk + 同步 | 流式 WebSocket + partial | **架构重构** |
| VAD | 无 | Silero VAD | 新增 |
| 翻译 | GLM-4-Flash | GLM-4-Flash（保留） | 不变 |
| 并发 | 单用户 | 3-5 session 隔离 | **新增 session 管理** |
| 幻觉防护 | 基本无 | 置信度过滤 + 重复检测 | 新增 |
| 部署 | 本地 CPU | GPU 服务器 | **需要 GPU** |

**最关键的改造**：
1. 从 chunk 改成流式
2. 模型从 medium 升到 large-v3-turbo
3. 加 VAD
4. 加 session 管理支持多用户
5. 需要 GPU

## 9. 我的推荐

### 第一步（验证期，1~2 周）
1. 搭 WhisperStreaming + large-v3-turbo 原型
2. 用真实场景音频（广播、采访、面对面模拟）做评测
3. 拿到真实数据：准确率、延迟、幻觉率
4. 决定是否继续这条路线

### 第二步（MVP 期，2~3 周）
1. 流式前后端联通
2. 3~5 人 session 管理
3. 基本可用的产品形态

### 第三步（生产化）
1. 监控、日志、告警
2. 降级策略
3. 用户反馈收集

## 10. 给波哥的一句话总结

> **免费路线可以做到"广播/采访/面对面交流基本可用 + 3~5人轻生产"。**
> **准确率预估 90%~96%，延迟 2~5s，需要一台 GPU 服务器（或高配 Mac）。**
> **但稳定 97% + 3s 以内，免费路线大概率做不到。**
> **建议先花 1~2 周搭原型验证，拿到真实数据再拍最终方案。**
